# 项目路线与阶段门

状态：Current

## 目标与当前位置

目标是建立一套完整、可计算、可纸面验证的任务驱动增量游戏框架，再通过开局与高潮两个端点确认它是否有趣。

当前处于阶段 1：完整系统框架。阶段 0 已完成；当前不写具体人物、资源名、事件剧情或代码。

当前任务是起草“完整系统框架 v0.1”，至少确定：时间与 tick、人口和岗位、资源节点与加工边、设施与几何成本、规则里程碑、自动化规则、任务与事件数据结构、统一的产速与期限公式、离线结算语义，以及数值回放输出。

## 阶段 0：项目重置（已通过）

目标：清除旧规格的当前效力，只保留增量机制研究。

通过依据：旧文档与原型已归档；当前入口不再引用旧内容；研究目录只保留增量算法和 Web 底座研究；新决策台账已建立。

## 阶段 1：完整系统框架（当前）

目标：先定义时间、人口、岗位、资源、加工、设施、成本、任务、事件、成长、自动化、离线和反馈之间的完整关系。

通过条件：

- 每个模块都有输入、输出和依赖。
- 资源更新顺序、动作结算、倍率组合和里程碑语义没有歧义。
- 任务期限能够进入产能、回本时间和投资判断。
- 在线与离线使用同一结算语义。
- 不借助具体人物或剧情也能解释循环为何成立。

## 阶段 2：开局纸面段

目标：验证玩家如何从第一次手动产出进入自动设施、人口岗位和第一条加工链。

通过条件：短时间内至少发生两次瓶颈迁移；第一项自动化接管玩家已经解决过的问题；产速、成本和预计完成时间始终可读；不依靠文本奖励也愿意继续增长。

## 阶段 3：高潮纸面段

目标：验证多层资源、多个岗位、自动化和高压任务同时存在时，玩家仍能理解局面并作出不同取舍。

通过条件：高潮不是单一总数阈值或随机结果表；任务紧迫性真实改变投资和库存策略；至少两条策略可以完成目标但留下不同后果；成功或失败能由生产历史解释。

## 阶段 4：完整过程拼接

目标：设计两端之间的解锁、任务节奏、瓶颈迁移和自动化升级。

通过条件：每次解锁都有前因和新用途；没有长期闲置资源；旧层逐步自动化；新层持续制造新瓶颈；中段不是重复开局或单纯拖时长。

## 阶段 5：完整数值模型

目标：让固定策略在可重复模拟的纯数值核心中跑完整程。

通过条件：至少四类策略脚本可以运行；没有负库存、永久死锁、无界漏洞或无法解释的产出；在线与离线结果一致；产速、库存、停工、回本时间和任务压力均可导出审计。

## 阶段 6：Web 技术底座

目标：在领域规则稳定后选择长期可维护的实现底座。

通过条件：模拟核心独立于界面；技术栈不强迫玩法服从固定威望树或 UI；存档迁移、测试和离线计算有明确方案；依赖可重复安装且许可证合规。

## 阶段 7：可玩初版

目标：实现开局、中段和高潮的完整流程，而不是几个孤立系统。

通过条件：任务和事件真实驱动资源调度；自动化减少重复操作但不替玩家决定新目标；真人试玩能够区分数值、内容和界面问题。

## 停止线

- 阶段 1 通过前，不进入具体内容设计和代码实现。
- 阶段 2、3 都通过前，不拼接完整过程。
- 完整数值模型稳定前，不确定正式 Web 技术栈。
