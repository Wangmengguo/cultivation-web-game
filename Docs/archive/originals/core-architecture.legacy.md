# 宗门修真 · 核心架构设计

撰写日期：2026-07-09
角色定位：本文档是项目的顶层设计文档，回答三个问题——底层逻辑、核心玩法、拓展方向。后续所有数值表、系统设计、原型迭代都应能追溯到本文档的某一条原则。

## 0. 一句话定位

**用 Progress Knight 的数值循环做骨架，用 Football Manager 的"排兵布阵 + 可归因演算"做决策深度，用开罗游戏的"三层反馈周期 + 低频关键决策"做体验节奏，题材语义全部落在宗门修仙上。**

玩家是宗主，从头到尾不操作任何角色。玩家做的每一件事都是"安排"：排班、分配资源、指定突破者、编成历练队伍。时间流逝后，玩家通过日志和状态变化验收自己安排的结果，归因，再调整。

三个对标各取什么、各弃什么：

| 对标 | 取 | 弃 |
| --- | --- | --- |
| Progress Knight | 多倍率互相放大的循环、解锁驱动的期待感、"转生后旧阶段飞快"的兑现 | 单角色视角、纯菜单呈现 |
| Football Manager | 多轴属性 + 隐藏信息、阵容/轮换决策、演算结果可归因 | 30+ 属性的复杂度、实时比赛引擎 |
| 开罗游戏 | 短/中/长三层反馈周期、每周期只有几类关键决策、"系统在工作"的可视化 | 像素小人美术依赖、浅资源链 |

## 1. 底层逻辑：数值架构

底层逻辑的目标是：**任何新模块都能以"资源节点 + 倍率来源 + 门槛条件"三件套插进系统，不改核心结算代码，且数值不失控。**

### 1.1 四种数值原语

整个游戏的状态只由四种原语构成：

1. **存量（Stock）**：库存资源、弟子修为、设施等级、传承点。会被消耗或重置。
2. **流量（Flow）**：每游戏日的产出/消耗速率。由统一结算公式计算。
3. **倍率（Multiplier）**：作用在流量上的系数。所有系统对外的"效果"最终都表达为倍率。
4. **门槛（Gate）**：解锁与晋级条件。硬门槛（集齐特定资源组合）+ 软速率（攒够修为的速度由倍率决定）。

### 1.2 统一结算公式（倍率总线）

每个弟子每个游戏日的产出走同一条公式：

```text
日产出(弟子 d, 岗位 j, 资源 r)
  = Base(j, r)                    # 岗位基础产出
  × Attr(d, j)                    # 弟子属性系数（资质/武力等映射）
  × (1 + Σ 功法加成)              # 同类内加法
  × (1 + Σ 设施加成)
  × (1 + Σ 供给加成)              # 丹药、灵食、洞府待遇
  × (1 + Σ 境界加成)              # 弟子自身境界带来的效率
  × (1 + Σ 传承加成)              # 大转生永久倍率（后期）
```

**叠加铁律：同类加法，跨类乘法。** 这是本项目最重要的一条数值规则，来自增量游戏社区共识与 Pecorella《The Math of Idle Games》的分析：纯乘法叠加（Progress Knight 的做法）实现简单但系统一多必然爆炸；同类先求和再跨类相乘，每一类都是一个可独立预算、可独立封顶的"倍率槽"。

工程上这就是"倍率总线"：核心结算只认注册在总线上的倍率类别，任何新系统（功法、设施、丹药、阵法、传承）上线时做两件事——向总线注册自己的倍率类别，声明自己消耗/产出哪些资源节点。核心循环代码永远不需要知道具体系统是什么。**"各模块用数值的方式交叉叠加"的工程表达就是这条总线。**

### 1.3 数值治理规则（防爆炸预算制）

每个系统在设计阶段必须先填"倍率预算表"，再写具体数值。治理规则：

1. **单个倍率来源随等级只允许线性或对数增长，禁止指数。** Progress Knight 源码的实际做法：`1 + log10(level+1)`（对数）、`1 + maxLevel/10`（线性）、`1 + 0.01×level`（弱线性）。这是它多系统纯乘法却不爆炸的原因。
2. **升级成本一律指数：`cost = base × rate^level`，rate 取 1.08~1.12。**（AdVenture Capitalist 1.07、Cookie Clicker 1.15，我们取中间偏低，前期节奏更顺。）指数成本 + 线性产出 = 自然形成"墙"，墙就是节奏点，不需要人为卡关。
3. **每类倍率设计上限，全局乘积设计上限。** 首发版本预算示例：

| 倍率类别 | 单项增长形态 | 类内上限（首发） | 类内上限（长线） |
| --- | --- | ---: | ---: |
| 弟子属性 | 静态系数（资质决定） | ×3 | ×10 |
| 功法 | 线性 `1 + 0.05×level` | ×5 | ×20 |
| 设施 | 线性 `1 + 0.1×level` | ×4 | ×15 |
| 供给 | 档位制（丹药/灵食分档） | ×2.5 | ×5 |
| 境界 | 每境界 ×1.5~×2 | ×4 | ×30 |
| 传承（大转生） | 对数 `1 + k×ln(1+p)` | 首发不做 | 不封顶但对数 |

全局乘积首发控制在 ×10³ 以内，长线控制在 ×10⁶~10⁹。超预算的设计一律打回。

4. **每个资源节点必须有至少一个上游和至少一个消耗端**（终端资源除外）。资源只进不出会永久过剩，只出不进会永久瓶颈。
5. **加工溢价 1.2~1.3：** 下游成品链的单位时间价值 = 上游原料链 × 1.2~1.3。奖励玩家参与加工，但不至于让采集岗位失去意义。
6. **瓶颈自检：** 数值表做好后跑模拟——任何资源库存若单调增长超过 3 个反馈周期，下调其上游或给下游加消耗槽；若持续为零，反向处理。

### 1.4 资源图（DAG）

资源链是一张有向图，首发版本刻意保持最小，但拓扑结构从第一天就按可扩展设计：

```text
弟子时间（唯一的真正稀缺资源）
  ├─ 采药 → 凡阶材料 ─┬─ 炼丹 → 丹药 ─→ 修为加速 / 突破消耗
  │                    └─ （后期）设施升级消耗
  ├─ 历练（山脚）→ 凡阶材料
  ├─ 历练（妖兽林，有武力门槛）→ 灵阶材料 ─→ 筑基突破消耗
  └─ 修炼 → 修为 ─→ 突破 ─→ 境界（跨类倍率）─→ 解锁更高级岗位/区域
```

设计不变量：**弟子·天 是唯一的原生稀缺资源，所有资源归根到底是弟子时间的变形。** 排班之所以是核心决策，就是因为它是在分配这个唯一稀缺物。后续加任何系统前先问：它消耗弟子时间吗？不消耗的话它凭什么占据玩家的决策带宽？

材料等阶（凡阶 → 灵阶 → 后续更高阶）是资源图的纵向扩展方式：更高进度要求"接触更好的材料来源"，而不是"同一种材料要更多"。这避免了数字通胀式的伪深度。

### 1.5 弟子：倍率的载体，决策的落点

弟子是连接所有系统的中心实体。数值上，弟子 = 一组属性系数 + 一个境界（跨类倍率）+ 一个岗位指针。

属性架构分三层，按阶段逐层开放：

1. **首发（已在 MVP）**：资质（修炼效率系数）、武力（历练门槛）、境界、修为。共 4 个可见量，全部直接可读。
2. **FM 化阶段**：扩展到约 8~12 个可见属性（悟性、体魄、心性、神识、灵根倾向等），每个属性必须至少挂钩一个岗位或一类事件，不允许"只是数字大"的装饰属性。**上限 16 个**——FM 敢用 30+ 是因为有几十年球迷文化共识兜底，我们没有。
3. **信息不对称阶段**：引入隐藏属性（潜力上限＝FM 的 PA、心魔倾向、道缘），通过"长老考察"给出带模糊度的报告。这是长线耐玩性的来源：看得见的属性 + 猜不透的化学反应，让每个弟子成为一个有性格的命题，而不是一行战力值。

### 1.6 双层转生结构

Prestige 是放置游戏长线动力的数学核心。本项目做两层：

**小转生：弟子突破（首发已有雏形）。**
- 突破消耗：满修为（硬速率门槛）+ 特定资源组合（硬资源门槛）。确定成功，无概率。
- 突破重置：修为归零。
- 突破保留：境界 +1，作为该弟子的跨类永久倍率（×1.5~×2），并给宗门一个小的全局加成。
- 体验目标：复刻 Progress Knight 的核心爽点——**上一轮很慢的东西，这一轮明显变快**。筑基弟子回头做炼气期的工作应该有碾压感。

**大转生：宗门传承（拓展期）。**
- 触发语义待定（候选：宗主坐化传位 / 宗门劫难后重建 / 开辟新道统），机制上是全局重置。
- 传承点公式采用平方根范式：`p = √(累计宗门总产值 / K)`。Pecorella 的对比数据：平方根曲线下传承点翻倍约需 4 倍增长（AdVenture Capitalist 范式），节奏适中；立方根（Cookie Clicker）翻倍需 8 倍，更硬核。我们取平方根起步。
- 传承点只兑换**线性或对数**永久倍率，绝不给指数倍率——这是"下一轮更快，但不会无限缩短"的数学保证（Progress Knight 的 evil 系统同理：受控正反馈，而非指数爆炸）。

### 1.7 时间模型

- 离散 tick：1 游戏日为最小结算单位（当前 12 秒/天，可调）。每日结算顺序固定：排班读取 → 生产结算 → 消耗结算 → 修为结算 → 门槛/解锁检查 → 日志写出。
- **确定性模拟**：同样的排班 + 同样的初始状态 = 同样的结果。首发版本零随机。随机性（事件、暴击产出）是后期调味品，且引入时必须保证"随机只影响幅度，不影响成败"——成败必须永远可预读（如"武力不足"提前显示），这是可归因性的底线。
- 离线 = 快进补结算，逐日跑同一套确定性模拟，产出压缩摘要日志。行业惯例：离线效率 50%~100%、时长上限 8~12 小时。**当前 MVP 的 100 天上限 ≈ 现实 20 分钟，作为放置游戏明显过短，需要在下一版重新标定**（见第 4 节遗留问题）。

## 2. 核心玩法

### 2.1 核心循环

```text
【排】排班调阵：给每个弟子指定岗位，指定优先突破者，选择历练目标
  ↓
【看】时间流逝：日志滚动，资源和修为同时增长，放置中有事发生
  ↓
【读】验收归因：从日志和状态变化读出"我的安排产生了什么效果、瓶颈在哪"
  ↓
【调】低频干预：换供给方案、调岗位、改优先级——随时可改，次日生效
  ↓
【破】集中兑现：攒够条件，让一名弟子突破 → 全局加速 → 解锁新内容
  ↓ 回到【排】，但这一轮明显更快、选项更多
```

这个循环同时满足三个体验承诺：
- **开罗式即时互动**：任何时候打开游戏都有日志在动、有决策可做，但没有任何操作需要反应速度。
- **FM 式调兵布阵**：决策的本体是"把有限的人放到哪个位置"，收益差来自人与岗位的匹配，而不是点击效率。
- **放置式长线**：关掉游戏，宗门继续运转；回来时验收的是"弟子状态变化"，不是一堆待领取的红点。

### 2.2 三类决策（FM 映射）

玩家的全部决策收敛为三类，对应 FM 的三个决策场景：

| 决策类型 | FM 对应 | 内容 | 频率 |
| --- | --- | --- | --- |
| 经济阵型 | 联赛轮换 | 谁生产、谁修炼、谁历练；材料链怎么配平 | 每几个反馈周期一次 |
| 培养路线 | 训练安排 | 优先突破者是谁；丹药/供给集中给谁；练谁的功法 | 每个突破周期一次 |
| 战术阵容（拓展期） | 比赛日首发 | 秘境派哪几人、什么站位/阵法、带什么保命符 | 每次历练一次 |

三类决策共享同一套弟子属性，但看重的属性不同——这正是多轴属性存在的意义：没有"最强弟子"，只有"这个位置最合适的弟子"。资质高的坐镇丹房还是闭关冲境界？武力高但资质平庸的弟子是历练主力还是浪费了他的岗位？**当玩家开始为这类问题纠结时，玩法就成立了。**

### 2.3 三层反馈周期（开罗映射）

| 周期 | 时长（现实时间） | 玩家看到什么 | 玩家决策什么 |
| --- | --- | --- | --- |
| 日 | 12 秒 | 日志逐条弹出、资源跳动、修为条推进 | 通常不决策，纯观察 |
| 旬/月 | 2~5 分钟 | 丹药成品、历练队回归、解锁提示、阶段小结 | 调排班、换供给 |
| 突破/年度 | 20~60 分钟 | 弟子突破大事件、新岗位/区域开放、（后期）宗门大比 | 培养路线、长线战略 |

节奏基准（来自增量游戏行业数据 + 对标验收标准）：
- **≤ 5 分钟**：第一次买得起效率提升（第一次兑现）。
- **≤ 15 分钟**：第一次解锁新岗位或功法（第二次兑现）。
- **30~60 分钟**：第一次突破，且突破后旧阶段的加速肉眼可见（第三次兑现）。
- **任何时刻**：下一个可见目标的达成时间 ≤ 当前收益速率的 3~10 倍。超过 10 倍，玩家会认为"没东西玩了"。
- 内容默认隐藏，接近条件时才浮现（Progress Knight 范式）——期待感来自"刚刚看见但还差一点"。

**验收标准沿用对标文档的表述：不是"循环能跑"，而是玩家至少经历三次明确兑现。三次兑现有一次不爽，先调数值节奏和反馈，不加新系统。**

### 2.4 可归因性：演算不许黑箱

FM 研究给出的最大教训：自动演算的策略游戏，死于黑箱。玩家看不懂"为什么输"，策略性就退化成玄学，然后流失。本项目的可归因性设计为硬性规范：

1. **每条改变状态的日志必须带括号数值**（已是 CONTEXT.md 规范）：`林青云把丹药当糖豆吃，进展很快（丹药 -1，修为 +12）`。
2. **失败必须给原因，且尽量提前预告**：历练前显示"能应付/武力不足"，而不是去了才失败。
3. **拓展期的历练战报必须有复盘**：不只给胜负和战利品，给 2~3 条具体归因（"甲的武力压制了妖狼""乙修为不足，全程被保护"），相当于 FM 的赛后分析 + 助教点评。
4. **倍率可查**：玩家能在弟子面板看到当前产出是哪几类倍率乘出来的。数值游戏的乐趣一半在"看懂自己的机器"。

### 2.5 惩罚模型：收益差，不是事故

首发版本坚持 CONTEXT.md 的既有决策：排班错误的后果是**产出效率低**，不是过劳、心魔、炸炉。负面事件（可失败的历练、心魔风险）属于拓展期内容，且引入时必须遵守 2.4 的可预读原则。先让玩家信任系统，再让系统偶尔咬人。

## 3. 拓展方向

分阶段推进，每个阶段有明确的验证问题，上一阶段验证不通过就不进入下一阶段。**每个新系统必须通过三问：消耗弟子时间吗？向倍率总线注册了什么？玩家多了哪个不可兼得的选择？**

### 阶段 0：MVP 原型（已完成）

单文件 `index.html`：5 弟子、4 岗位、3 境界、确定突破、离线补结算。
验证问题：排班 → 观察 → 突破的基础节奏是否成立。**当前状态：待自测三次兑现是否爽。**

### 阶段 1：数值骨架完整化（Progress Knight 化）

目标：把"多倍率互相放大"完整跑起来。

- **功法系统**：每个弟子同时修一门功法（对应 PK 的技能），功法等级提供线性倍率，挂到倍率总线。修炼岗产修为，功法给全岗位效率——形成"生产 vs 变强"的第一个张力。
- **设施系统**：丹房、灵田、静室，等级用 `base × 1.08^level` 指数成本，材料消耗接入资源图，倍率线性。
- **解锁链**：岗位、功法、设施、历练区域全部条件解锁、默认隐藏。
- **离线重标定**：确定天/秒比例与离线上限（小时级），离线摘要日志。
- 验证问题：玩家会不会为"先升设施还是先攒突破材料"纠结？墙出现的位置和频率是否舒服？

### 阶段 2：调兵布阵深化（FM 化）——本项目的差异化核心

- **属性扩展**：可见属性扩到 8~12 个，每个属性挂钩至少一个岗位/事件。
- **招收弟子**：不定期出现候选弟子（FM 的转会窗），信息带模糊度（长老考察报告），宗门规模上限制造取舍。**不做抽卡**——竞品研究已明确这条路被《最强祖师》占满，且会把设计拖向养成堆料。
- **历练队伍编成**：外出历练从单人门槛检查升级为多人队伍——选人、定位（前排/后排/辅助）、配保命符。自动演算 + 战报复盘。这是"调兵布阵"从经济阵型扩展到战术阵容的关键一步。
- **中途低频干预**：历练过程中允许 1~2 次高杠杆干预（撤退令、保命符），有代价、有冷却。这是 FM 临场换人的移植，也是"即时互动但不吃操作"的上限所在。
- 验证问题：玩家排队伍时是否在看属性匹配而不是战力总和？战报复盘能否让失败被理解？

### 阶段 3：宗门生命感（开罗化）

- **事件系统**：访客求丹、灵田虫灾、邻宗来访、弟子悟道。事件只改资源和小状态，用 2~3 选项决策，不做剧情树。
- **年度节点**：宗门大比 / 外门考核，给中长周期一个可预期的锚点和排行反馈。
- **可视化升级（可选，按成本决定）**：优先级为 弹出数字 > 日志丰富度 > 小人走动 > 场景进化。前两者纯 UI 成本，已可支撑"系统在工作"的生命感；小人走动是开罗招牌但依赖美术，放最后。
- 验证问题：不打开面板、只看主场景，玩家能否感到"宗门活着"？

### 阶段 4：长线循环（大转生）

- **宗门传承**：全局重置 + 传承点（√ 公式）+ 对数永久倍率。传承语义候选：宗主传位、劫后重建、开辟道统——需要一次专门的叙事/机制讨论。
- **新周目差异**：传承解锁新灵根、新功法流派、新地图区域，让多周目是"换个build再来"而不是"同样的事再做一遍"。
- 验证问题：第一次大转生时，玩家是期待还是心疼？（心疼说明保留物设计不足。）

### 阶段 5+：明确不做清单（来自竞品研究，作为长期护栏）

- ❌ 多人宗门战 / 跨服社交（《一念逍遥》的领地，会把设计拖向战力运营）。
- ❌ 《了不起的修仙模拟器》级深模拟（内容与 QA 成本不可承受）。
- ❌ 抽卡弟子养成（《最强祖师》的领地）。
- ❌ 互不相干的"百艺"按钮合集（没有资源互锁的系统堆叠 = 同质化面板 idle）。

我们要守住的空位是竞品地图上明确存在的：**可读的资源链 + 可观察的宗门 + FM 式的人事决策**，这三者的交集目前没有成熟产品。

## 4. 待讨论的开放问题

以下决策影响架构走向，需要共同确认后再进入对应阶段：

1. **历练的最终形态**：阶段 2 把历练升级为队伍编成 + 自动演算 + 复盘，是"调兵布阵"的主要落点。同意这个定位吗？还是希望战斗味更淡、始终停留在门槛检查？
2. **时间尺度与离线策略**：12 秒/天是验证用参数。正式节奏需要定：一个突破周期多少现实分钟？离线上限几小时？这决定它是"每天看三次的手游节奏"还是"挂后台的网页节奏"。
3. **大转生语义**：宗主传位 / 劫后重建 / 开辟道统，三个候选的叙事味道差别很大，影响玩家对"重置"的情绪接受度。
4. **技术路线**：对标文档提出 Blazor WebAssembly 重写，当前原型是单文件 HTML。建议：数值验证期（阶段 0~1）留在单文件快速迭代，阶段 2 属性系统复杂化之前再做一次技术选型，届时重新评估 Blazor vs 轻前端框架 vs 继续原生。
5. **弟子死亡/流失**：FM 的老化与转会是其长线张力来源。修仙语义里对应寿元与叛门/夺舍。首发明确不做，但阶段 2 引入招收弟子时，"只进不出"的宗门会通胀——需要预设弟子的退出机制。
