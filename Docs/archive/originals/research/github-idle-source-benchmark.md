# GitHub 放置游戏源码对标

研究日期：2026-07-09

## 结论

主对标选 [`Progress Knight`](https://github.com/ihtasham42/progress-knight)。

不是因为它和宗门经营最像，而是因为它最适合回答当前最重要的问题：一个几乎没有美术的网页放置游戏，怎样只靠数值、解锁和选择形成持续爽点。

它具备以下条件：

- 已发布到 [GitHub Pages](https://ihtasham42.github.io/progress-knight/)，并在 Armor Games、CrazyGames 上发布。
- GitHub 当前有 172 stars、131 forks，明显高于同类修仙题材实验仓库。
- 源码只有 8 个主要文件，本地检出约 248 KB，适合完整读懂后迁移。
- 使用 [Unlicense](https://github.com/ihtasham42/progress-knight/blob/main/LICENSE)，允许复制、修改、发布和商业使用。
- 核心数据、成长公式、解锁条件和 UI 更新都集中在 [`js/main.js`](https://github.com/ihtasham42/progress-knight/blob/main/js/main.js) 与 [`js/classes.js`](https://github.com/ihtasham42/progress-knight/blob/main/js/classes.js)，不依赖游戏引擎。

## 它的爽点结构

`Progress Knight` 不是靠单一数字增长，而是让多个增益相互放大：

1. 玩家同时选择一个工作和一个技能，工作提供收入，技能提供长期倍率。
2. 收入允许承担更高生活成本，高级住所和物品又提高经验倍率。
3. 工作、技能升级后解锁更高级工作、技能和物品。
4. 角色死亡后重生，当前等级和资产重置，但历史最高等级形成经验倍率。
5. 下一轮会明显快于上一轮，玩家能快速重新跨过曾经困难的门槛。

源码中的关键机制：

- 每项任务有等级、当前经验和升级所需经验。
- 升级成本随等级增加，但历史最高等级提供永久经验倍率。
- 工作等级提高收入，技能等级提高其他系统的效率。
- 新内容默认隐藏，只有满足条件后才出现。
- 收入和持续支出同时存在，玩家必须在成长速度和经济承受能力之间调整。

它的循环可以概括为：

```text
选择工作与训练
→ 获得收入和经验
→ 购买更高倍率的生活方式
→ 解锁更高阶工作与技能
→ 本轮成长放缓
→ 重生并获得永久倍率
→ 更快跨过旧门槛，进入新内容
```

真正的爽点是“上一轮要等很久的东西，这一轮迅速完成”，而不是离线回来领取一堆资源。

## 如何转换成宗门修仙

第一步不创新循环，只替换语义并保留原有反馈节奏：

| Progress Knight | 宗门版本 |
| --- | --- |
| 当前工作 | 弟子岗位 |
| 当前技能 | 修炼功法 |
| 收入 | 宗门材料产出 |
| 生活方式 | 洞府、丹房和修炼供给 |
| 物品倍率 | 丹药、法器和长老指点 |
| 工作晋升 | 岗位升阶、历练区域解锁 |
| 角色重生 | 境界突破 |
| 历史最高等级倍率 | 突破后的修炼效率与宗门传承 |

第一轮创新只增加一项：把单角色变成三名资质不同的弟子。每名弟子仍然只同时执行一个岗位和一种修炼，宗主决定资源集中给谁。

不要立即增加地图、战斗、装备随机词条、建筑摆放和大量事件。先证明以下基础节奏成立：

```text
排班
→ 看到资源和修为同时增长
→ 更换供给方案提高倍率
→ 解锁新岗位或功法
→ 集中资源让一名弟子突破
→ 突破后旧阶段明显加速
```

## 备选源码及拒绝理由

### [`Idle Loops`](https://github.com/dmchurch/omsi-loops)

它的动作队列、自动执行和永久成长最接近“排班后观察”，玩法上非常值得研究。但主流 GitHub 分支没有明确许可证，不能直接复制代码。

可借鉴机制：

- 玩家先安排动作队列，再观察整轮自动执行。
- 每轮失败点直接暴露下一个瓶颈。
- 永久能力让下一轮能够执行更多动作、走到更远区域。

### [`wrtsc`](https://github.com/deifactor/wrtsc)

MIT 许可的 `Idle Loops` 衍生实现。它把任务队列、资源需求、任务奖励和预测器拆成独立游戏逻辑，并用同一套模拟器预测排程结果。

它适合作为后续“排班预测”实现参考，但项目仍处于 beta，只有第一地区基本完成，不能承担主对标的玩法验证责任。

### [`Evolve`](https://github.com/pmotschmann/Evolve)

活跃、内容丰富、MPL-2.0 许可，擅长资源链、人口岗位和逐层解锁。但本地检出约 28 MB、超过 100 个源码文件，已经是成熟长线游戏。直接对标会再次把第一版做重。

### [`Gooboo`](https://github.com/Tendsty/gooboo)

有完整测试、移动端支持和多个增量系统，但使用 GPL-3.0，且本地检出有 500 多个文件。适合研究后期系统组织，不适合当前原型。

### [`CultivationIdle`](https://github.com/shlocko/CultivationIdle)

题材直接相关，包含修炼、冥想、历练、战斗、随机事件和离线修为，但只有 1 star，仓库没有许可证，且本地检出约 193 MB。题材相同不代表玩法经过验证，也不能合法复制代码。

## Blazor WebAssembly 工程参考

[`dotnet-blazor-games`](https://github.com/ZacharyPatten/dotnet-blazor-games) 使用 MIT 许可，展示了 Blazor WebAssembly 游戏组件和 GitHub Pages 部署。

它只作为工程参考，不作为玩法对标。仓库的部署工作流展示了三个关键点：

- `dotnet publish --configuration Release`
- 将 `<base href="/">` 改成 GitHub Pages 仓库子路径
- 发布目录加入 `.nojekyll`

该仓库使用旧版 .NET 5 工作流，后续应按当前 .NET 10 官方模板重新建立，不直接复制旧工作流。

## 推荐执行方式

1. 先完整体验 `Progress Knight` 的第一轮和第一次重生，记录每次产生期待的解锁点。
2. 把它前 20 至 30 分钟的工作、技能、支出、解锁和重生曲线整理成数值表。
3. 用 Blazor WebAssembly 重写同等规模的宗门版本，不移植 JavaScript 代码结构。
4. 第一版只做三名弟子、四个岗位、四种供给、一次确定突破。
5. 验收标准不是“循环能跑”，而是玩家是否至少经历三次明确兑现：
   - 第一次买得起效率提升。
   - 第一次解锁更高阶岗位或功法。
   - 第一次突破后明显加速。

如果这三个时刻仍然不爽，先改数值节奏和反馈，不加新系统。
