## 对话提出的完整开发流程

### 在代码之前写可验证的游戏说明

先明确玩家身份、主要动作、一轮如何开始和结束、成功与失败、单次时长，以及第一版只实现哪一段完整体验。建议用 `GAME_SPEC.md` 保存这些内容，重点是每项都可以测试。

### 根据游戏类型选择技术

2D 动作、卡牌或棋盘类可以考虑 Phaser；普通 3D 可以考虑 Three.js；经营、放置、策略和文字占比高的游戏可以用普通 Web UI、Blazor、React 或 Vue。技术应由玩法和界面需求决定，不是所有游戏都需要游戏引擎。

### 分离状态、规则、渲染、输入和界面

游戏状态保存会影响结果的数据；规则系统处理数值和状态转移；渲染只表现状态；输入转成动作；复杂菜单和文本界面优先使用 DOM。规则不应只存在于 Canvas、精灵或动画对象中。

### 建立稳定的项目目录和进度记录

对话建议把应用启动、领域状态、规则系统、命令、事件、渲染、输入、界面、资源、存档、调试和测试分开，并使用 `progress.md` 记录需求、完成项、测试结果、问题和下一步。

### 先做一段完整的 Vertical Slice

第一版应很窄，但从输入、规则、反馈、结果到重开全部接通。可以使用临时图形和少量内容，不能只做许多孤立系统再尝试拼接。

### 为自动测试提供状态和时间接口

对话建议提供 `window.render_game_to_text`，用简短 JSON 输出当前可见、可交互且影响决策的状态；同时提供 `window.advanceTime`，让测试能够确定性推进时间，而不是依赖真实等待。这些接口可以只在开发或测试构建中启用。

### 每次只改一个可验证行为

推荐循环是：选一个目标，做最小改动，启动游戏，模拟输入，推进状态，检查截图、文本状态和控制台，再修正。验证需要覆盖完整因果链，不能只看最终画面。

### 测试分为三层

规则单元测试验证数值和状态转移；系统集成测试验证多个模块连接后的事件和结果；浏览器测试验证真实输入、流程、不同尺寸和视觉表现。Canvas 或 WebGL 项目还需要截图检查。

### 图片和其他资源需要稳定流程

角色动画应先确定基础帧，再生成统一比例和锚点的完整动画条带。代码通过稳定资源键引用资产，避免把临时文件名散落在规则代码中。

### 用完成标准约束功能

一个功能需要同时满足规则实现、测试通过、真实输入可触发、画面与文本状态一致、控制台无新增错误、相关失败与重开流程可用、桌面和手机尺寸可操作、进度文档已更新。发布前再进行构建、端到端测试和部署环境检查。
