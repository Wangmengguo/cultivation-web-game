## 总裁决

**文档的核心设计原则是对的，尤其是回答了你现在最关心的那个争议：交互性应该来自"排班 + 资源集中 + 指定优先突破者"，而不是平均培养或自动喂丹。** 这条在文档里不是空话，而是有具体机制支撑的（1.5 节"优先突破者"资源决策、2.5 节"收益差不是事故"），也和 `CONTEXT.md` 里明确写的 `_Avoid_: 平均培养, 自动最优培养` 一致。这是全文档最该保留、也是新雏形第一个要验证的东西。

但文档里混杂了三类不同性质的内容，你在"基于它搭新雏形"时必须先分开处理：
- **(a) 设计不变量**（长期有效，应保留）
- **(b) 旧 MVP 的具体参数**（伪装成设计结论，实际只是占位符，应丢弃重设计）
- **(c) 一条已过时的技术路线建议**（与 `HANDOFF.md` 记录的最新决定直接冲突，需要你现在拍板）

## 1) 文档中正确、应保留的原则

- **一句话定位 + 三对标取舍表**（0 节）：方法论扎实，三个对标各自"取什么弃什么"写得很克制，没有贪多。
- **弟子·天是唯一稀缺资源**（1.4 节）：这是整个"排班为什么是核心决策"的数学基础，也是回应你争议的关键——只要每个新系统都要先问"它消耗弟子时间吗"，游戏就不会滑向自动喂丹式的挂机。
- **同类加法、跨类乘法 + 倍率总线**（1.2、1.3 节）：这是增量游戏防爆炸的行业共识，方向正确。**但要注意**：这是一条"数值组织规则"，不是"必须现在写的代码架构"。`HANDOFF.md` 里明确说第一个可玩切片不要加接口、DI、倍率插件系统——这和 core-architecture.md 的表述并不矛盾，但容易被误读成"第一版就要搭一套可插拔总线"。第一版应该用最朴素的硬编码算式满足"同类加、跨类乘"这条规则即可，不需要注册机制。
- **确定突破、收益差、可归因性**（1.6、2.4、2.5 节）：三条互相支撑——突破无概率、排班错误只是效率低不是事故、每条日志带括号数值——这是"玩家能看懂自己的机器"的底线，也是 FM 式策略游戏不死于黑箱的关键。
- **三层反馈周期 + 三次兑现验收标准**（2.3 节）：给了可测试的验收线（5 分钟/15 分钟/30-60 分钟），比"循环能跑"这种模糊标准可执行得多。
- **阶段化推进 + 每阶段一个验证问题 + 明确不做清单**（3 节）：护栏画得清楚（不做抽卡、不做深模拟、不做跨服），这是抵抗过度设计最有效的部分。

## 2) 已落后 / 被旧 MVP 绑架的内容

- **"阶段 0：MVP 原型（已完成）"这个状态已经过期**——`index.html` 已被你否决删除（`HANDOFF.md` 记录"no clear payoff, weak UI, no satisfying progression loop"），文档还按"已完成"叙述，需要更新，否则新雏形会误以为阶段 0 的结论已经验证过。
- **具体参数是旧原型的实现细节，不是设计结论**：4 个岗位（修炼/采药/炼丹/历练）、3 境界（炼气/筑基/金丹）、"12 秒/天"、"100 天离线上限"——这些数字全部来自 `mvp-prototype-plan.md`，文档 1.7 节自己也承认"当前 MVP 的 100 天上限……需要在下一版重新标定"。**新雏形不应该照搬这些数字**，它们只是被验证过一次"跑得起来"但已经被判定"不好玩"的参数组合，沿用它们等于把旧原型的失败原样搬进新版。
- **技术路线建议与最新决定冲突**：文档 4.4 建议"数值验证期留在单文件快速迭代，阶段 2 前再评估 Blazor"。但 `HANDOFF.md` 把这条标记为"superseded"，改为直接上 Standalone Blazor WebAssembly + .NET 10 + GitHub Actions 部署。**这是一个需要你现在拍板的真实冲突**：你这次找我要的是"快速迭代雏形"，而 Blazor 路线需要装 SDK、脚手架、CI 部署，这些都是和"快速验证手感"无关的一次性成本。我的建议见下方裁决。
- **数值治理的具体倍率上限表**（1.3 节表格：功法×5、设施×4、全局×10³……）属于阶段 1+ 才需要的精细化产物，第一版雏形不需要，也不应该现在就设计预算表——先验证交互核心好不好玩，倍率预算是第二步的事。这部分本身没写错，但**放在"第一版要做什么"里读会误导人**，需要明确标注"阶段 1 以后再启用"。

## 3) 第一版新雏形应该验证什么

只验证一件事，别的都先不管：

> **玩家通过"排班 + 把有限资源集中给指定的一个弟子（优先突破者）+ 等待确定性结算"这套决策，是否比"平均分配"或"系统自动帮你决定最优"更让人有参与感、更让人纠结？**

拆成可观察的子问题：
1. 排班（谁去哪个岗位）产生的资源效果差异是否**当场可读**（不用翻表就能看懂谁该调岗）？
2. "优先突破者"这个集中资源的决定，是否让玩家真的犹豫（"给这个人还是那个人"），而不是显然的最优解？
3. 三次兑现节奏（≤5min 第一次效率提升 / ≤15min 第一次解锁 / 30-60min 第一次突破且旧阶段明显变快）是否成立？

**不需要在第一版验证**：多属性匹配、战报复盘、事件系统、传承——这些都是文档已经标注的"拓展期"内容，验证核心决策之前碰它们纯属浪费。

## 4) 最小功能集

- 3~5 名资质/武力不同的弟子，每人一个岗位指针（不需要 FM 化属性，4 个可见量：资质/武力/境界/修为够用）
- 3~4 个岗位：至少 1 个纯产出（采药）、1 个转化（炼丹）、1 个修炼、历练可选（如果想更快验证核心决策，历练甚至可以先砍掉，只留"生产 vs 修炼"这一组张力）
- 唯一的核心机制：**"优先突破者"标记 + 确定性突破规则**（满修为 + 材料门槛，无概率）——这是全部要验证的东西，必须做到位
- 资源图砍到最小：1 种材料 → 1 种加工品（丹药）→ 修为加速。不需要凡阶/灵阶两级，不需要历练支线，除非你想同时验证材料等阶这个子设计
- 时间模型：确定性 tick + 快进按钮 + 简单离线补结算即可，天/秒比例和离线上限先随便定一个能测试的数（比如 5-10 秒/天），**不用现在标定成正式节奏**——这是文档自己说的"待讨论问题"，属于拿到手感反馈之后再调的参数
- 日志面板作为验收中心，括号数值必须有，语气可以先随便写
- **不需要**：功法系统、设施系统、倍率总线的注册机制、数值预算表——这些等第一个核心决策被验证"好玩"之后再加

## 5) 必须后移、避免过度设计的系统

- 功法/设施系统的完整倍率总线抽象（阶段 1 内容，先用硬编码系数代替）
- 数值治理的倍率预算表、指数成本曲线精细标定（阶段 1）
- FM 化的 8~16 个可见属性、隐藏属性/长老考察（阶段 2、阶段 2+）
- 招收弟子机制（阶段 2）
- 历练队伍编成 + 自动演算 + 战报复盘、中途干预（撤退令/保命符）（阶段 2）
- 事件系统、年度节点、场景可视化升级（阶段 3）
- 宗门传承/大转生（阶段 4）

这些文档自己已经归类到位，唯一要提醒的是：**不要因为文档把"功法系统"放在阶段 1（听起来像"下一步"）就误以为它该出现在第一版雏形里**——阶段 1 的前提是阶段 0 的核心决策已经验证成立，而阶段 0 现在需要重新做一次（旧的被否决了）。

**技术路线**：如果目标真的是"快速迭代雏形"来验证手感，我的建议是维持 `core-architecture.md` 4.4 节的原议——单文件 HTML/JS 快速迭代，把 Blazor 决定推迟到阶段 2 前。`HANDOFF.md` 记录的 Blazor 决定（装 .NET 10 SDK、脚手架、GitHub Actions 部署）是一次性工程成本，和"这一版是否好玩"这个问题无关，会拖慢你现在最需要的验证速度。如果你已经笃定要 Blazor（比如出于长期工程考虑），那就不必现在纠结，但这和"快速迭代雏形"的诉求本身有张力，值得你明确决定一次，而不是让两份文档的冲突悬着。
