## 总裁决

**架构本体是对的，且水平不低。** 四数值原语、倍率总线、叠加铁律、预算制治理、确定性可归因、双层转生数学、不做清单——这套东西来源清楚（Pecorella / Progress Knight / FM / 开罗）、内部自洽、工程上能防爆炸，是少见的"想清楚了再写"的设计文档。**问题不在"对不对"，而在"锚点错了"**：文档把一个已被否决的 index.html 当成"阶段0 已完成"的验证基线，并据此安排 staging 和技术路线，这一点与 `HANDOFF.md` 的 Blazor 决定直接冲突。所以结论是：**保留架构，重锚 staging，雏形只做阶段0 内核。**

## 逐问裁决

### Q1 — 玩家作为宗主的"下令/资源分配权"能否成为核心？

**能，定位成立。** 但有两个文档低估的薄弱点，必须在雏形里验出来，否则"核心"会塌成"数字在涨"。

**成立的部分（保留）：**
- "玩家只安排、不操作角色"是真实品类定位（Progress Knight / Melvor / FM 都验证过"安排→看→归因→调"是闭环），且是竞品空位（可读资源链 + 可观察宗门 + FM 式人事决策的交集确实没成熟产品）。
- **"弟子·天是唯一原生稀缺资源"是全文最值钱的一条不变量。** 它让排班真正有重量——你不是在"点按钮"，你是在分配唯一稀缺物。这一条足以支撑"下令/资源分配"成为核心。
- 三类决策（经济阵型 / 培养路线 / 战术阵容）映射干净，给了决策多样性。

**两个薄弱点（雏形必须验，不是架构错）：**
1. **循环中段"看→读"在零随机下很薄。** 文档把所有随机/事件推到阶段3，但 `CONTEXT.md` 的"放置中有事发生"是**首发承诺**（avoid: 空等/一键收菜）。零随机的确定性模拟，"有事发生"很容易退化成"数字在涨 + 一次突破"。一个 30–60 分钟的雏形里，纯确定性日志第二次读就没新鲜感了——这是"低频管理游戏"最致命的地方。**保留零随机原则，但雏形必须实测：只看确定性日志，半小时内你是否还有事可读。**
2. **"优先突破者=核心资源决策"只在供给相对多弟子足够稀缺时才成立。** 如果首发只有一名弟子需要突破就"通关"雏形，最优解就是"全员喂一个"，毫无取舍，"核心资源决策"就是空话。取舍是否存在，**完全取决于数值调参**（文档 1.3 rule 4/6 已经给了自检手段，但首发数值没定）。**雏形必须实测：你在"给谁喂丹药、谁去历练"上是否真的犹豫过。**

> 裁决：核心定位对，"弟子·天"锚点对。但核心成不成立，不由架构优雅度决定，而由上面两点在雏形里能否兑现决定。

### Q2 — 文档是否仍受旧 index.html / MVP 影响？

**是，且影响不止 4.4 一处。** `HANDOFF.md` 的冲突条款已经明确把 4.4 的"阶段0~1 留单文件"判为 superseded；但 MVP 痕迹其实更广：

| 位置 | MVP 痕迹 | 问题 |
|---|---|---|
| §3 阶段0 | "已完成 / 单文件 index.html / 待自测三次兑现" | index.html 已被用户否决（无兑现、UI 弱、循环不爽），它**不是已验证基线**，是已废弃产物。把废品当基线，后续 staging 全错位。 |
| §1.5 / §1.6 | "首发（已在 MVP）" / "首发已有雏形" | 措辞继承了一个已不存在的基础。 |
| §1.7 | "当前 12 秒/天" / "当前 MVP 的 100 天上限 ≈ 现实 20 分钟" | 这两个数是 MVP 派生量，文档自己也承认"过短需重新标定"——不应作为"当前值"继承到新雏形。 |
| §4.4 | "数值验证期留单文件，阶段2 再选型" | 与 Blazor 决定直接冲突，已被 HANDOFF 否决。 |

**关键判断：MVP 影响的是 staging 和技术路线，不是架构本体。** §0–§2 的底层逻辑和核心玩法基本与 MVP 无关、是站得住的；§3 的阶段划分和 §4.4 才是被 MVP 污染的重灾区。所以处理方式是**重锚 staging、改写 4.4、去"已在 MVP"措辞、把时间参数降级为待标定**，而不是动架构本体。

### Q3 — 哪些原则保留，哪些改？

**保留（架构核心，勿动）：**
- 玩家=宗主、只安排不操作
- 四数值原语（存量/流量/倍率/门槛）
- **倍率总线 + "同类加法、跨类乘法"叠加铁律**（全文最重要的一条）
- 数值治理预算制（线性/对数倍率 + 指数成本 + 每类上限 + 全局乘积上限）
- **弟子·天 = 唯一原生稀缺资源**（设计不变量）
- 资源图 DAG + 加工溢价 + 瓶颈自检
- 确定性模拟 + 可归因性（括号数值/失败给原因/倍率可查）
- 收益差不是事故（首发无惩罚，先赢信任）
- **三次兑现验收标准**（不是"循环能跑"，而是"3 次兑现至少 1 次爽"——这是防系统膨胀的正确闸门）
- 双层转生数学（小转生=突破；大转生=√公式 + 对数永久倍率）
- 不做清单（无抽卡/无多人宗门战/无深模拟）+ 新系统三问门

**改 / 重锚：**
- §3 阶段0：从"已完成 (index.html)"改为"已否决，需在 Blazor 上重建雏形"。
- §3 阶段1：不再表述为"在 MVP 上补全数值骨架"，而与雏形一起从零搭；且**阶段1 的功法+设施+解锁链不应进雏形**（见 Q4）。
- §4.4：改写为与 Blazor 决定对齐，删掉"留单文件"建议。
- §1.5/§1.6：去掉"已在 MVP"措辞，统一为"首发版本"。
- §1.7：12 秒/天、100 天上限从"当前值"降级为"待标定起始猜测"。

**保留但在雏形里重点验（不改原则、验风险）：**
- 零随机 ⟷ "放置中有事发生" 的张力
- "优先突破者=核心资源决策" 的取舍是否真实

### Q4 — 第一版雏形验证什么、最小功能、不要过度设计

**雏形 = 阶段0 内核，在 Blazor WASM 上从零重建，不是文档的阶段1。** 文档阶段1 把"功法+设施+解锁链+离线重标定"捆成一整块，对"快速迭代雏形"是过度设计——这些是**内核验证通过之后**才加的东西。

**雏形要验证的 5 件事（任一失败=核心不成立，先调数值不加系统）：**
1. 排班→看→归因→调→突破 这个循环 30–60 分钟内不无聊。
2. 三次兑现按时发生（≤5min 第一次效率提升 / ≤15min 解锁新岗位或历练区 / 30–60min 第一次突破且旧阶段肉眼变快），且**至少一次爽**。
3. "集中供给优先突破者"构成真实取舍（你真的犹豫过给谁）。
4. 可归因性：你能指着日志说出"卡在灵阶材料，因为妖兽林武力门槛没到"。
5. 放置有事发生：关掉再回来，补结算摘要你想看，不是一键收菜。

**最小功能（对齐 HANDOFF 的最小架构，落到具体）：**

```
GameState
  day
  disciples[5]: { id, name, 资质, 武力, 境界(首发3级), 修为, job }
  stocks: { 凡阶材料, 灵阶材料, 丹药 }
  priorityDiscipleId       // 恰好一名
  expeditionTarget         // 山脚 | 妖兽林(武力门槛)
  unlocks                  // 岗位/区域可见位

GameRules (纯函数, 与 Razor 解耦, 在线/快进/离线/预告同源)
  AdvanceOneDay(state) -> DayResult
    读排班 → 生产 → 消耗 → 修为 → 门槛检查 → 日志(每条带括号数值)
  CanBreakthrough(state, d) -> { ok, blocking[] }   // 满修为? 灵阶材料够? 提前预告
  Breakthrough(state, d) -> { state', logs }         // 确定成功, 修为归零, 境界+1, 解锁一个新岗位/历练区/全局小加成

SaveStore
  localStorage 版本化 JSON; 离线=逐日跑 AdvanceOneDay + 压成摘要, 上限提至小时级

UI (单页 operational surface)
  顶: day / 速度(1x·暂停) / 三资源 / 当前突破目标+卡点
  左: 弟子名册(资质/武力/境界/修为条/岗位)
  中: 时间流逝日志(括号数值) + 进度
  右: 排班控件 + 预期产出 + 瓶颈说明 + 突破按钮(带预告)
  移动端: 单列, 目标+运营在前
```

**关键反过度设计点：** 倍率总线**只接两类**（资质静态系数 × 境界跨类倍率），保留总线**结构**但不填 6 类预算表。日产出首发即 `Base(job,res) × 资质系数 × (1+境界倍率)`，到此为止。

**明确不做（过度设计红线）：**
- ❌ 功法 / 设施 / 完整解锁链（文档阶段1，验证后再加）
- ❌ 8–12 属性 / 隐藏属性 / 长老考察（阶段2）
- ❌ 历练队伍编成 / 战报复盘（阶段2；首发历练=单人门槛检查）
- ❌ 事件 / 年度节点 / 小人走动（阶段3）
- ❌ 大转生 / 传承（阶段4）
- ❌ DI / 接口层 / 仓储 / 事件总线 / 可复用倍率插件系统（HANDOFF 已禁，此处重申）
- ❌ 倍率预算 6 类全实现（只接资质+境界）

**失败时的唯一动作：调数值**（供给稀缺度 / 突破成本 / 天秒比 / 离线上限），**不加系统**——这是文档自己定的闸门，也是雏形阶段最重要的纪律。
