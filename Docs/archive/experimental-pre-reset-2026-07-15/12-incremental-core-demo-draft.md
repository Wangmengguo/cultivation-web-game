# 宗门增量核心 Demo 初稿 v0.1

## 一、这次究竟验证什么

本 Demo 只验证一个承重命题：

> 玩家只看数字、岗位卡和资源流，通过重配弟子、投资设施、打通加工链和完成突破，是否能持续感到“宗门正在被我构筑得越来越会运转”。

现有纸面与阶段三模型已经证明：具名人物、寿元、稀缺资源和公开期限可以形成可解释取舍；它们没有证明“生产—转化—岗位重配—升级—自动化”本身好玩。详见 [当前原型缺口审计](research/current-prototype-gap-audit-2026-07.md)。

因此本轮暂不同时验证完整世代、人物自主作息、战斗姿态、随机突破、复杂伤势和动态小人。传承只保留一个结构性预告，危机只承担一次公开、确定性的能力验收。

## 二、推荐方向

### 核心循环

```text
观察库存、净速与耗尽时间
→ 找到当前瓶颈
→ 重配弟子岗位或改变资源用途
→ 建设设施 / 加工丹药 / 培养弟子
→ 产速、转化率或个人能力发生台阶跃迁
→ 旧环节被自动化，出现新的瓶颈
→ 用已经构筑出的宗门能力通过赤潮验收
→ 赤潮从威胁转为自动资源点
```

这一结构组合了三组已有一手证据支持的机制：

- [Kittens Game](https://wiki.kittensgame.com/en/general-information/resources) 与 [Evolve Idle](https://github.com/pmotschmann/Evolve) 的离散人口/岗位分配；
- [Melvor Idle](https://wiki.melvoridle.com/index.php?title=Beginners_Guide) 的原料—加工—用途跨技能链和按完整动作结算；
- [Antimatter Dimensions](https://store.steampowered.com/app/1399720/Antimatter_Dimensions/) 的几何成本、里程碑倍增和逐层自动化。

完整证据、固定源码提交与许可证边界见 [已验证的增量游戏数值机制研究](research/validated-incremental-algorithms-2026-07.md)。这些作品验证的是机制骨架，不证明宗门版本天然好玩；本 Demo 仍需验证岗位冲突、瓶颈迁移和修仙语义。

## 三、玩家能控制什么

首版只给玩家五种控制权：

1. 把四名弟子分配到采集、炼丹、修炼或待命；每人同时只能做一件事。
2. 决定灵材用于经济设施、丹药加工还是护山阵。
3. 决定丹药培养哪一名弟子，不允许系统平均分配。
4. 决定何时让达到门槛的弟子突破；突破期间该弟子停止工作。
5. 赤潮到来时决定哪些弟子驻守；驻守者暂时离开生产岗位。

玩家暂时不能控制：小人走路、吃饭、逐招战斗、攻击目标、战斗姿态、随机事件选项和跨代传承分配。

## 四、最小资源与岗位模型

### 资源图

```text
采集弟子 ──→ 灵材 ──┬──→ 探采堂 / 丹炉 / 聚灵阵
                     ├──→ 护山阵
                     └──→ 丹房动作 ──→ 丹药 ──→ 指定弟子修为 ──→ 突破

赤潮胜利 ──→ 自动狩猎点 ──→ 持续灵材
```

首版只有两种共享库存资源：灵材、丹药；修为属于具体弟子，不是公共货币。这样既能形成加工链，又不会一开局就出现完整修仙资源树。

### 初始弟子

| 弟子 | 采集适性 | 炼丹适性 | 修炼适性 | 基础防御 | 首版作用 |
|---|---:|---:|---:|---:|---|
| 岳徊 | 1.50 | 0.80 | 0.80 | 12 | 最强基础资源通路 |
| 沈砺 | 0.90 | 1.50 | 0.80 | 10 | 最强加工者 |
| 归元 | 0.70 | 0.80 | 1.50 | 30 | 高防御、高突破价值 |
| 林砚 | 0.80 | 0.90 | 1.30 | 8 | 可培养的未来主力 |

这些角色沿用现有纸面资产，但适性和防御数字是本轮测试参数，不继承旧表格中的临时战力系数。

## 五、首轮算法

### 1. 连续采集

```text
灵材产速 = 0.5 × Σ采集弟子适性 × (1 + 0.15 × 探采堂等级) + 自动狩猎产速
```

单位为灵材/秒。界面同时显示库存、净速、距离下一设施所需时间和按当前消耗预计耗尽时间。

### 2. 炼丹按完整动作结算

```text
丹房动作能力 = Σ炼丹弟子适性 × (1 + 0.15 × 丹炉等级)
基础动作时长 = 10 秒
每个完整动作消耗 6 灵材，产出 1 丹药
```

每个 tick 累积丹房工作进度；只有进度达到一次完整动作且原料足够时才扣料出丹。原料不足时停止并显示“缺少灵材”，不得让库存为负，也不得按比例凭空出丹。

### 3. 修炼按具体弟子结算

```text
个人修炼动作能力 = 修炼适性 × 境界倍率 × (1 + 0.20 × 聚灵阵等级)
基础动作时长 = 5 秒
每个完整动作消耗 1 丹药，获得 5 修为
```

达到 100 修为后可开始突破。首版突破确定成功、持续 30 秒、不额外消耗资源；突破期间停止原岗位。成功后该弟子的境界倍率由 1.0 变为 2.0，基础防御额外增加 30。这里先验证“是否值得为台阶跃迁暂时损失产能”，不让随机成功率污染结论。

### 4. 设施成本

常规设施采用成熟增量游戏反复使用的几何成本：

```text
下一等级成本 C(n) = ceil(C0 × 1.15^n)
```

其中 `n` 是当前等级。

| 设施 | C0 | 每级效果 | 3 级里程碑 |
|---|---:|---|---|
| 探采堂 | 35 | 采集倍率 +15% | 灵材容量 +200，并显示各采集者贡献拆分 |
| 丹炉 | 40 | 丹房动作能力 +15% | 每完成 5 次炼丹，第 5 次额外产出 1 丹药 |
| 聚灵阵 | 50 | 修炼动作能力 +20% | 每完成 5 次修炼，第 5 次不消耗丹药 |

平滑等级负责等待期间的小成长，3 级里程碑负责改变规则。成熟作品常在更长流程里使用 5/10 等里程碑；这里压缩到 3 级只为适配 15—20 分钟 Demo，必须通过试玩重新校准。

护山阵单独使用更陡的扩容成本：

```text
护山阵下一等级成本 = ceil(80 × 1.25^n)
每级提供 20 防御
```

### 5. 产速倍率的组合规则

同一来源类别内先相加，不同类别再相乘：

```text
岗位产速 = 基础值 × Σ弟子适性 × 设施倍率 × 境界倍率 × 里程碑倍率
```

首版不加入品质暴击、随机稀有掉落、天气、心境和全局软上限。每一个倍率都必须能在 UI 中展开，玩家可以追溯“为什么现在是 1.73/秒”。

## 六、赤潮只做确定性能力验收

赤潮在第 12 分钟到来，提前全程显示倒计时、需求和当前缺口。

```text
宗门防御 = 20 × 护山阵等级 + Σ驻守弟子当前防御
胜利门槛 = 100
驻守持续 = 30 秒
```

- 达到 100：确定胜利，解锁“赤潮狩猎点”，自动提供 0.5 灵材/秒；曾经的威胁成为生产链的一部分。
- 未达到 100：损失当前灵材的 20%，赤潮在 60 秒后重整；玩家可以调整后重试，不形成不可恢复死局。

该设计产生两条明确但可混合的准备路线：投入护山阵以保留更多生产弟子，或投入丹药和时间完成关键突破，用更少驻守者达到门槛。首版删除 A/B/C/D 概率表和攻守姿态；它们只有在能力构筑本身成立后才有资格重新进入。

## 七、15—20 分钟体验节奏目标

| 时间段 | 期望发生的事情 | 观察指标 |
|---|---|---|
| 0—1 分钟 | 玩家第一次重配岗位 | 调人后 5 秒内能解释净速变化 |
| 1—4 分钟 | 第一次设施投资，瓶颈从灵材迁移到丹房或丹药 | 玩家是否主动改变岗位，而不是照提示点击 |
| 4—8 分钟 | 第一个设施里程碑或弟子突破 | 是否出现明显的产速台阶，而非只感到等待变短 |
| 8—12 分钟 | 经济投资与赤潮准备发生冲突 | 是否存在至少两条可辩护路线，不出现唯一机械答案 |
| 12—14 分钟 | 赤潮确定性验收 | 玩家能否在结算前准确预测胜负与代价 |
| 14—16 分钟 | 狩猎点自动产出，旧压力被转成新产能 | 玩家是否明确感到“宗门会自己处理旧问题了” |
| 16—20 分钟 | 显示下一层预告并结束本轮 | 玩家是否愿意继续构筑，而不只是想看剧情 |

上述是调参目标，不是已经验证的完成时间承诺。

## 八、技术底座初稿

### 推荐：受控 fork Profectus 0.7，但设置硬门槛

Profectus 0.7 是本轮唯一同时提供 Vue 3、TypeScript、Vite、大数、资源、Formula、Requirement、Conversion、Repeatable、Modifier、Layer、本地多存档、离线追赶、测试与 PWA 的候选。对应关系如下：

| 宗门概念 | Profectus 原语 |
|---|---|
| 灵材、丹药、个人修为 | Resource / persistent ref |
| 炼丹与修炼 | 自定义 Action + 输入约束；不直接使用 prestige reset |
| 设施等级 | Repeatable + Formula + Requirement |
| 产速解释 | Modifier |
| 突破 | Action + Requirement |
| 宗门阶段 | Layer，但界面不使用默认威望树 |
| 存档升级 | `fixOldSave` 迁移入口 |

但是官方模板当前不能原样视为稳定底座。本地干净安装实测因 Galaxy 云存档 Git 服务返回 502 而失败，且模板只声明 Node 21。采用前必须完成以下硬门槛：

1. 受控 fork 后移除 Galaxy 云存档耦合，仅保留本地存档、导入和导出；
2. 锁定 Node 与依赖版本，在干净环境完成安装、构建和测试；
3. 宗门模拟核心使用纯 TypeScript，不依赖挂载 Vue 才能运行；
4. 自定义一页宗门调度台，不显示默认威望树，不靠覆盖大量框架 CSS 才能成立；
5. 对在线 1 分钟、离线 1 分钟和离线 1 小时写确定性对照，库存、动作余量和修为必须符合定义；
6. 如果多资源链因框架 tick 顺序产生不可接受偏差，立即停止继续迁就框架。

完整框架比较与许可证见 [Web 增量/放置游戏底座调研](research/web-incremental-frameworks-2026-07.md)。

### 失败退路

若 Profectus 任一硬门槛失败，退路固定为：Vue 3 + Vite + 独立纯 TypeScript 模拟核心 + `break_eternity.js` 或 `emath.js`。这会多写存档迁移、离线切片和批量购买基础设施，但不会退回 TMT、旧 IGT，也不会为纯数字 Demo 引入 Phaser。

## 九、数值验证方法

首轮不靠“玩起来感觉差不多”判断稳定性。模拟器至少输出以下记录：

- 每秒各资源库存、净速、来源和消耗；
- 每个岗位实际工作时间、原料不足停工时间；
- 每项设施的购买时点与实际回本时间；
- 每次岗位调整前后 30 秒的产速差；
- 突破前后个人产能和宗门防御差；
- 赤潮前 60 秒的经济投入、防御投入和最终驻守人数；
- 在线与离线同操作脚本的终局差异。

必须跑四类策略脚本：全经济、全防御、优先突破、混合路线。通过条件不是四者结果相等，而是：

1. 没有资源负数、无限增长漏洞或永久停工死锁；
2. 至少两条路线能通过赤潮，且机会成本不同；
3. 全经济不能在最后一刻无代价补齐防御；
4. 全防御能安全但明显更慢，不能与混合路线等价；
5. 优先突破能减少驻守劳动力，但为此真实牺牲前期加工或建设；
6. 每种资源在解锁后始终至少有两个竞争用途；
7. 玩家调整一名弟子后，5—10 秒内能看到可解释反馈。

## 十、明确不做

- 不把阶段三概率表迁入新 Demo；
- 不做三种战斗姿态；
- 不做小人场景移动和完整作息；
- 不做随机突破、伤势和死亡；
- 不做第二种危机和完整传承周目；
- 不复制 Kittens Game 源码，其 WET PAWS LICENSE 禁止商业衍生；
- 不承诺 Profectus 已经正式选定，只有通过上述硬门槛后才锁定。

## 十一、最脆弱假设

本方案假设：**仅靠岗位重配、资源链和台阶式能力跃迁，数字/卡片呈现就足以产生持续乐趣。**

如果真人试玩仍只是在执行显而易见的最优步骤，说明问题不是内容少，而是资源网络没有产生真正的构筑空间。此时应调整岗位互斥、资源用途和里程碑规则，而不是先加入人物故事、随机事件或动画掩盖它。

## 十二、需要确认的推荐结论

建议确认以下方向后再实现：

1. 首个 Demo 采用“多弟子并行岗位资源网络”，不是 Progress Knight 式单角色单线成长；
2. 首轮突破与赤潮完全确定性，随机性延后；
3. 赤潮胜利后直接转化为自动资源点；
4. 技术上先以受控 fork Profectus 0.7 为候选，必须先解决可重复构建、UI 解绑和离线一致性，失败则切换到 Vue 3 + 纯 TypeScript 模拟核心。
